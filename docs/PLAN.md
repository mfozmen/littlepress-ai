## Next up

Items below came out of the first real end-to-end test (Yavru Dinozor). Listed roughly in "most visible to the user" order.

- **`<MIXED>` text-only default survives ingestion but is overwritten by `propose_layouts`.** Reported 2026-04-25 after the user rendered ``yavru_dinozor``. The draft.json shows pages 1–4 with ``layout=image-top``/``image-bottom``/``image-full`` and ``image`` attached — exactly the shape PR #67's fix was meant to prevent. Root cause: ingestion correctly classifies MIXED and sets ``layout=text-only``, but the agent then proactively calls ``propose_layouts`` after its greeting (the tool is still registered + its docstring actively encourages rhythm work). ``propose_layouts`` sees image-attached pages, applies rhythm rules, and promotes them to ``image-*`` layouts — undoing the MIXED default. Fix: make ``propose_layouts`` treat ``layout=text-only AND image is not None`` as a protected shape (the unique signature of a MIXED-default page, since non-MIXED text-only has no image). User can still override manually via ``choose_layout``. Needs a regression test that pins: after a MIXED-classified ingestion + ``propose_layouts`` call, the affected pages stay text-only.

- **Illustrations can't be extracted from Samsung Notes / phone-scan pages — text and drawing are baked into the same pixels.** Reported 2026-04-25 same session. User's framing: *"yazı ve resmi tek bir resim olarak eklemeye devam ediyor. resmi çıkartamıyor. ilusstrasyonları yazılardan ayırt edemiyor yani."* This is an architectural gap, not a bug. The pipeline treats each Samsung Notes page as one raster — text-pixels and drawing-pixels intermixed with no metadata separating them. MIXED classification + ``text-only`` default hides the image but loses the drawing; ``image-top`` shows the image but the text prints twice (once baked in, once transcribed). Fix options: **(a) OCR-box masking + local inpainting** — vision returns word bounding boxes (tighten the OCR prompt or switch to tesseract for boxes), mask those regions in the image, inpaint with a neighbour-fill algorithm (OpenCV ``cv2.inpaint`` or similar); cheap, no API cost, quality varies with handwriting bleed into the drawing. **(b) OpenAI gpt-image-1 edit endpoint** — pass the page image + a "remove the handwritten text, preserve the drawing" prompt; simple, costs ≈ $0.06–$0.25 per page, quality depends on the prompt. **(c) Workflow change** — document the limitation and ask the user to provide drawing and text separately (photo of paper drawing + typed text) when they want clean illustrations; loses Samsung Notes convenience but is the only pixel-perfect solution. **(d) Offer ``generate_page_illustration`` as a replacement** — when MIXED classification defaults a page to text-only, suggest the existing AI illustration tool as an opt-in way to get a new drawing for that page; tool already ships and the user has OpenAI access. Likely path: (d) as the near-term lever (no new infra), (a) as a follow-up experiment when someone has time for the image-processing work.

- **Colophon pages must be detected and routed to cover metadata, not rendered as story pages.** Reported 2026-04-25. User's page 5 was a Samsung Notes colophon reading ``YAZAR:POYRAZ RESİMLEYEN:POYRAZ``; it was transcribed correctly and rendered as a regular text-only interior page, but it should have been recognised as metadata: the author name folded into ``cover.author`` (merged with the user-typed full name, ``Poyraz Özmen``) and the page hidden from the story. User's framing: *"kapata olmalıydı bu yazılar ve poyraz özmen olarak yazmalıydı. yapay zeka bunları ayırt etmeli."* This is judgment work — exactly the kind of task the "AI-only-for-judgment" principle preserves. Sub-project 2 removed the agent from upfront metadata collection but didn't add a ingestion-time "classify this transcribed page's purpose" pass. Fix shape: after OCR classification, an extra agent-turn looks at each transcribed text block and asks "is this story content or book metadata (colophon / credits / dedication / copyright)?" For pages classified as metadata: (i) auto-hide, (ii) merge any author / illustrator names it finds with the existing cover metadata fields (prefer user-typed full name over OCR fragment — so ``Poyraz Özmen`` beats ``POYRAZ``, but a full name in the OCR beats an empty user-typed field). Touches ``src/ingestion.py`` (new colophon-detection pass, gated behind active provider — NullProvider skips) and ``src/metadata_prompts.py`` (name merger when colophon-extracted name is a superstring / substring of user-typed name).

- **Deterministic metadata prompts feel cold and English-only.** Reported 2026-04-25 same session, right after the Ctrl+C bug. The user was typing Turkish to the REPL but the five deterministic prompts (``Title?`` / ``Author?`` / ``Is this book part of a series?`` / ``Cover?`` / ``Back-cover blurb?``) print as single-word English labels regardless of the user's language. The agent-driven flow that Sub-project 2 replaced used to match the user's language automatically; the refactor dropped that warmth. The user's exact framing: *"sanırım ai sorsa soruları daha iyiydi."* Their concern isn't with determinism per se — it's that the refactor conflated "don't burn LLM tokens on data collection" with "terse English-only labels are fine." Fix options: (a) full localisation — auto-detect user language from the first reply and re-prompt subsequent questions in it, matching the agent's old multi-language behaviour; (b) config-driven — a ``--lang`` CLI flag + ``/lang`` slash command that sets the prompt language for the session; (c) hybrid — ship English + Turkish strings as a start (the only two the maintainer has used in practice) with auto-detect from the first reply, expand to a translations dictionary later. (c) is the smallest real improvement that gets the warmth back for the maintainer's own workflow without building a full i18n system yet. In any shape, phrasing should be full sentences ("What's the title of the book?" / "Kitabın adı ne?") rather than single-word labels, and cover/back-cover menus should have descriptive option text rather than bare ``(a)/(b)/(c)`` letters. See memory feedback ``determinism_is_not_english_only``.
