## Next up

Items below came out of the first real end-to-end test (Yavru Dinozor). Listed roughly in "most visible to the user" order.

- **`<MIXED>` text-only default survives ingestion but is overwritten by `propose_layouts`.** Reported 2026-04-25 after the user rendered ``yavru_dinozor``. The draft.json shows pages 1–4 with ``layout=image-top``/``image-bottom``/``image-full`` and ``image`` attached — exactly the shape PR #67's fix was meant to prevent. Root cause: ingestion correctly classifies MIXED and sets ``layout=text-only``, but the agent then proactively calls ``propose_layouts`` after its greeting (the tool is still registered + its docstring actively encourages rhythm work). ``propose_layouts`` sees image-attached pages, applies rhythm rules, and promotes them to ``image-*`` layouts — undoing the MIXED default. Fix: make ``propose_layouts`` treat ``layout=text-only AND image is not None`` as a protected shape (the unique signature of a MIXED-default page, since non-MIXED text-only has no image). User can still override manually via ``choose_layout``. Needs a regression test that pins: after a MIXED-classified ingestion + ``propose_layouts`` call, the affected pages stay text-only.

- **Cover override bug: agent calls ``set_cover`` even when REPL already set the cover.** Reported 2026-04-26 live render of a Samsung-Notes draft. User picked option (c) (poster, typography only) at the cover prompt; the REPL correctly set ``cover_image=None, cover_style="poster"``. The agent then ran ``read_draft``, saw ``cover_image=None + cover_style="poster"`` and inferred the cover wasn't yet selected, then called ``set_cover`` with a default (``full-bleed`` from a page drawing) — overriding the user's deterministic choice. The rendered book's cover is wrong. Root cause: the greeting says "metadata is already set by the REPL" but doesn't surface the SPECIFIC cover choice. Agent reads the draft state but can't distinguish "user chose poster (typography-only)" from "cover not configured yet" because both have ``cover_image=None``. Fix: ``_build_agent_greeting`` injects an explicit cover-state line per branch — ``cover_choice == "poster"`` → "Cover is set to poster style (typography only); do NOT call set_cover." Same for ``"page-drawing"`` (image already attached). Existing AI cover branch instructions stay. Needs regression tests in ``tests/test_agent_greeting.py`` for both non-AI branches.

- ~~Illustrations can't be extracted~~ **SHIPPED**: ``src/drawing_extraction.py::extract_drawing_region`` uses row-density analysis (numpy) to find the largest content block on a single-raster page (the illustration) and crop it cleanly. ``apply_sentinel_result``'s MIXED branch now wires the extraction in — when it succeeds, ``page.image`` swaps over to the cleaned ``page-NN.drawing.png`` and the layout becomes ``image-top`` so the rendered book gets a clean drawing under the transcribed text (no duplicate). Falls back to the old ``text-only`` default when extraction can't isolate a clear region (e.g. genuinely overlapping text-and-drawing pixels). End-to-end regression in ``tests/test_yavru_dinozor_e2e.py`` (skipped when the user's actual fixture isn't on disk; the PDF stays out of the public repo by ``.gitignore`` ``*.pdf`` rule) pins the full pipeline against the real Samsung Notes input. This is an architectural gap, not a bug. The pipeline treats each Samsung Notes page as one raster — text-pixels and drawing-pixels intermixed with no metadata separating them. MIXED classification + ``text-only`` default hides the image but loses the drawing; ``image-top`` shows the image but the text prints twice (once baked in, once transcribed). Fix options: **(a) OCR-box masking + local inpainting** — vision returns word bounding boxes (tighten the OCR prompt or switch to tesseract for boxes), mask those regions in the image, inpaint with a neighbour-fill algorithm (OpenCV ``cv2.inpaint`` or similar); cheap, no API cost, quality varies with handwriting bleed into the drawing. **(b) OpenAI gpt-image-1 edit endpoint** — pass the page image + a "remove the handwritten text, preserve the drawing" prompt; simple, costs ≈ $0.06–$0.25 per page, quality depends on the prompt. **(c) Workflow change** — document the limitation and ask the user to provide drawing and text separately (photo of paper drawing + typed text) when they want clean illustrations; loses Samsung Notes convenience but is the only pixel-perfect solution. **(d) Offer ``generate_page_illustration`` as a replacement** — when MIXED classification defaults a page to text-only, suggest the existing AI illustration tool as an opt-in way to get a new drawing for that page; tool already ships and the user has OpenAI access. Likely path: (d) as the near-term lever (no new infra), (a) as a follow-up experiment when someone has time for the image-processing work.

- **Colophon name merging into cover metadata.** PR for the auto-hide half of the colophon problem shipped (``feat/colophon-auto-hide``); a metadata page like ``YAZAR:POYRAZ`` no longer renders as an interior story page. The remaining slice from the original report: when the colophon-extracted name is a SUPERSTRING of the user-typed name (user typed ``Poyraz``, OCR saw ``Poyraz Özmen``), prefer the longer form on the cover. Today the user-typed value always wins. Defer until a real user run shows the merge case actually mattering — the user's primary complaint was the extra interior page, which auto-hide fixes; the cover already shows whatever the user typed.

- **Continue the "AI-only-for-judgment" refactor.** Sub-projects 1 and 2 shipped (see Shipped / `feat/deterministic-ingestion` and `feat/deterministic-metadata-collection`). Remaining:
  - **Sub-project 3 — (optional) Review-turn polish.** Possibly route review corrections through slash commands as well so the LLM's NL parsing is an explicit opt-in. Out of scope unless a real user test shows the free-form review is still too loose.

- **Spine-wrap cover template.** Five templates ship now (`full-bleed`, `framed`, `portrait-frame`, `title-band-top`, `poster`). The one remaining idea is `spine-wrap` — drawing spans front + spine + back for the A4 imposed booklet. This needs multi-page cover rendering support that the current `draw_cover` (single page) doesn't have; defer until a real user asks for it.
- **More image providers for AI cover generation.** First-slice ships OpenAI `gpt-image-1` (see "Shipped" below). Stability / Replicate / a local Stable Diffusion daemon are all plausible follow-ups — plug them in behind the existing `ImageProvider` protocol and add a user-visible way to pick.
